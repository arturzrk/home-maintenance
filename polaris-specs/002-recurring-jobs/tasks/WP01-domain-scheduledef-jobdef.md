# WP01 - Domain: ScheduleDefinition + JobDefinition + Job extension

## Objective

Add the three new Domain building blocks for recurring jobs: `CadenceUnit`
enum, `ScheduleDefinition` value object (with pure occurrence computation),
`StepTemplate` child entity, `JobDefinition` aggregate root, and a nullable
`JobDefinitionId` on the existing `Job` aggregate. No Application, Infrastructure,
or API code is touched in this WP.

### T002 [P] - ScheduleDefinition value object + OccurrencesInRange

Create `backend/src/HomeMaintenance.Domain/JobDefinitions/ScheduleDefinition.cs`.

Shape (from data-model.md):
- `CadenceUnit Unit` (required)
- `int Multiplier` (>= 1; throw `ArgumentOutOfRangeException` if < 1)
- `DateOnly StartDate` (required)
- `DateOnly? EndDate` (optional; if set, must be strictly > StartDate; throw `ArgumentException`)
- `sealed record` (value semantics)

Method `OccurrencesInRange(DateOnly from, DateOnly to)` returns `IEnumerable<DateOnly>`:

```
for n = 0, 1, 2, ...:
  occ = NthOccurrence(n)
  if occ > to or occ > EndDate: break
  if occ >= from: yield occ
```

Private helper `NthOccurrence(int n)`:
```csharp
Unit switch
{
    CadenceUnit.Day   => StartDate.AddDays(n * Multiplier),
    CadenceUnit.Week  => StartDate.AddDays(n * Multiplier * 7),
    CadenceUnit.Month => StartDate.AddMonths(n * Multiplier),
    CadenceUnit.Year  => StartDate.AddYears(n * Multiplier),
};
```

Note: `DateOnly.AddMonths` in .NET already handles month-end clamping (e.g.
Jan 31 + 1 month = Feb 28/29). No manual clamp needed.

Also stop when `n * Multiplier` would overflow or loop forever: add a max
iteration guard (`n > 10_000`) as a safety net.

### T004 - JobDefinition aggregate root

Create `backend/src/HomeMaintenance.Domain/JobDefinitions/JobDefinition.cs`.

Inherits `Entity` (same base class as `Job`). Fields:
- `OwnerId Owner` (immutable)
- `string PropertyId` (immutable, validated non-empty)
- `string Name` (non-empty, trimmed, max 200 chars)
- `List<StepTemplate> _stepTemplates` (private backing field)
- `ScheduleDefinition Schedule`

Public surface:
- `IReadOnlyList<StepTemplate> StepTemplates => _stepTemplates.AsReadOnly()`
- `static JobDefinition Create(string id, OwnerId owner, string propertyId, string name, ScheduleDefinition schedule, IEnumerable<string> initialDescriptions)`
- `internal static JobDefinition Hydrate(...)` (same params + steps list)
- `void Rename(string newName)` - validates name, trims
- `void UpdateSchedule(ScheduleDefinition schedule)` - replaces schedule
- Step template mutations returning outcome enums (same pattern as Job):
  - `StepTemplateMutationOutcome AddStepTemplate(string description)`
  - `StepTemplateMutationOutcome RemoveStepTemplate(string id)`
  - `ReorderStepTemplatesOutcome ReorderStepTemplates(IReadOnlyList<string> orderedIds)`
  - `StepTemplateMutationOutcome EditStepTemplateDescription(string id, string description)`

Define outcome enums in the same file:
```csharp
public enum StepTemplateMutationOutcome { Success, StepTemplateNotFound }
public enum ReorderStepTemplatesOutcome { Success, WrongCount, DuplicateId, UnknownId }
```

`AddStepTemplate` assigns Order = `_stepTemplates.Count` and generates a new
GUID id. `RemoveStepTemplate` calls `RenumberStepTemplates()` after removal.
`ReorderStepTemplates` validates all ids present then reorders, then renumbers.

- `JobDefinitionTests`: test each outcome enum variant at least once.

## Risks

- **Month-end clamping**: `DateOnly.AddMonths` handles it correctly in .NET.
  Verify with the Feb 29 leap-year test.
- **Infinite loop guard**: the `n > 10_000` safety net in `OccurrencesInRange`
  is paranoia for Day/Multiplier=1 over large ranges. It is fine for personal
  scale (3-month horizon = max 92 daily occurrences).
- **Naming collision**: `StepTemplateMutationOutcome` and `StepMutationOutcome`
  are separate enums in different namespaces - do not merge them.
